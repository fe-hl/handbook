一、LangGraph 概述 ​

1.1 什么是 LangGraph ​

LangGraph 是 LangChain 官方推出的 低层编排框架与运行时，用于构建：

  • 长流程 Agent
  • 有状态工作流
  • 可恢复执行的任务系统
  • 带人工介入的智能流程
  • 复杂多步骤推理系统

根据 LangGraph 官方文档，LangGraph 的核心定位是：

  • long-running
  • stateful
  • durable
  • agent orchestration

重点：LangGraph 不是 Prompt 工具库，而是 Agent 与工作流的编排框架。

1.2 LangGraph 适合什么 ​

LangGraph 特别适合这些场景：

  • 多节点 Agent 工作流
  • 需要显式状态流转
  • 需要暂停、恢复执行
  • 需要人工审批
  • 需要持久化上下文
  • 需要长生命周期任务

1.3 LangGraph 官方强调的核心能力 ​

LangGraph 官方 Overview 和 Durable Execution 文档反复强调的核心能力包括：

  • Durable execution
  • Persistence
  • Streaming
  • Human-in-the-loop
  • Stateful workflows

1.4 LangGraph 和 LangChain 的关系 ​

这个关系一定要讲清楚：

  • LangChain：更高层，适合快速构建 Agent
  • LangGraph：更底层，适合编排复杂状态流和工作流

而且 LangChain 官方 1.0 文档明确说明：

  • LangChain 的 Agent 构建在 LangGraph 之上

重点：如果你做的是复杂 Agent 系统，最终往往会落到 LangGraph。

二、为什么需要 LangGraph ​

2.1 只用 LangChain Agent 不够的场景 ​

如果你只做一个简单 Agent，LangChain 已经够用了。

但如果你开始出现这些需求，LangGraph 就很重要：

  • 需要多步决策流
  • 需要显式节点和边
  • 需要状态持久化
  • 需要恢复中断任务
  • 需要人工审批
  • 需要复杂分支逻辑

2.2 LangGraph 解决的核心问题 ​

LangGraph 主要解决：

  • 状态怎么存
  • 流程怎么编排
  • 中断后怎么恢复
  • 多步流程怎么持久化
  • 人工介入后怎么继续

2.3 一个典型 LangGraph 场景 ​

text
用户问题
 -> 意图分类
 -> 是否检索知识库
 -> 是否调用工具
 -> 是否需要人工审批
 -> 最终生成答案

这类流程如果只靠普通 Agent，很快会失控，而 LangGraph 特别适合这种显式编排。

三、LangGraph 核心概念 ​

3.1 Graph ​

Graph 就是整个工作流。

它由：

  • 节点（Node）
  • 边（Edge）
  • 状态（State）

组成。

3.2 Node ​

Node 是图中的一个执行单元。

一个节点通常负责：

  • 调模型
  • 调工具
  • 做分类
  • 改状态
  • 决定下一步

3.3 Edge ​

Edge 用来连接节点，表示流程怎么走。

3.4 State ​

State 是 LangGraph 的核心。

它表示当前流程运行到这一刻的上下文数据。

例如：

  • messages
  • classification
  • tool_result
  • user_id
  • approved

重点：LangGraph 的本质就是“节点 + 边 + 状态”的工作流系统。

3.5 START 与 END ​

LangGraph 提供两个特殊节点：

  • START
  • END

它们用来表示工作流入口和结束点。

3.6 Thread ​

根据 LangGraph 官方 Persistence 文档：

  • Thread 是状态持久化的主键

它通常由：

  • thread_id

来标识。

一个线程对应一条工作流执行历史。

3.7 Checkpoint ​

Checkpoint 就是某一步执行时保存的状态快照。

作用：

  • 支持恢复执行
  • 支持人类介入
  • 支持调试和时光回溯

3.8 Store ​

Store 用于保存跨线程、跨会话的长期信息。

它和 Checkpointer 的区别是：

  • Checkpointer 保存线程内执行状态
  • Store 保存跨线程长期记忆

3.9 Interrupt ​

Interrupt 是 LangGraph 里最关键的能力之一。

它可以让流程：

  • 执行到一半暂停
  • 等待外部输入
  • 之后从断点继续

3.10 LangGraph 核心关系图 ​

text
State
 -> Node
 -> Edge
 -> Checkpointer
 -> Thread
 -> Interrupt
 -> Store

四、安装与第一个 Graph ​

4.1 安装 ​

根据 LangGraph 官方文档：

bash
pip install -U langgraph

4.2 Python 版本 ​

LangGraph 当前官方 Python 文档体系与 LangChain 1.0 一样，建议使用：

  • Python 3.10+

4.3 Hello World 示例 ​

这段代码来自 LangGraph 官方 Overview 的主流思路。

python
from langgraph.graph import StateGraph, MessagesState, START, END


def mock_llm(state: MessagesState):
    # 返回新的 AI 消息
    return {"messages": [{"role": "ai", "content": "hello world"}]}


graph = StateGraph(MessagesState)
graph.add_node("mock_llm", mock_llm)
graph.add_edge(START, "mock_llm")
graph.add_edge("mock_llm", END)

app = graph.compile()

result = app.invoke({
    "messages": [{"role": "user", "content": "hi!"}]
})

print(result)

4.4 这段 Hello World 在做什么 ​

它做了四件事：

  1. 定义状态类型
  2. 定义一个节点
  3. 把节点接到图里
  4. 编译并运行

4.5 LangGraph 的基本使用步骤 ​

通常就是：

  1. 定义 State
  2. 定义节点函数
  3. 定义边
  4. compile
  5. invoke / stream

五、StateGraph 与节点编排 ​

5.1 自定义 State 示例 ​

python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages


class MyState(TypedDict):
    # 消息状态，add_messages 会自动追加消息
    messages: Annotated[list, add_messages]
    classification: str
    answer: str

5.2 多节点图示例 ​

python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END


class State(TypedDict):
    question: str
    classification: str
    answer: str


def classify_node(state: State):
    question = state["question"]
    if "天气" in question:
        return {"classification": "weather"}
    return {"classification": "general"}


def answer_node(state: State):
    if state["classification"] == "weather":
        return {"answer": "这是天气相关问题。"}
    return {"answer": "这是普通问题。"}


builder = StateGraph(State)
builder.add_node("classify", classify_node)
builder.add_node("answer", answer_node)
builder.add_edge(START, "classify")
builder.add_edge("classify", "answer")
builder.add_edge("answer", END)

graph = builder.compile()
result = graph.invoke({"question": "今天上海天气怎么样？"})
print(result)

5.3 条件边示例 ​

LangGraph 非常适合做条件分支。

python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END


class State(TypedDict):
    question: str
    route: str
    result: str


def router_node(state: State):
    if "RAG" in state["question"]:
        return {"route": "rag"}
    return {"route": "chat"}


def rag_node(state: State):
    return {"result": "进入 RAG 流程"}


def chat_node(state: State):
    return {"result": "进入普通聊天流程"}


def route_fn(state: State):
    return state["route"]


builder = StateGraph(State)
builder.add_node("router", router_node)
builder.add_node("rag_node", rag_node)
builder.add_node("chat_node", chat_node)
builder.add_edge(START, "router")
builder.add_conditional_edges(
    "router",
    route_fn,
    {
        "rag": "rag_node",
        "chat": "chat_node",
    },
)
builder.add_edge("rag_node", END)
builder.add_edge("chat_node", END)

graph = builder.compile()

5.4 Command 是什么 ​

LangGraph 中，Command 常用于：

  • 控制恢复执行
  • 在中断后继续
  • 指定下一步流转

5.5 消息状态图示例 ​

python
from langgraph.graph import StateGraph, MessagesState, START, END


def chatbot(state: MessagesState):
    user_msg = state["messages"][-1]["content"]
    return {
        "messages": [
            {"role": "ai", "content": f"你刚刚说的是：{user_msg}"}
        ]
    }


builder = StateGraph(MessagesState)
builder.add_node("chatbot", chatbot)
builder.add_edge(START, "chatbot")
builder.add_edge("chatbot", END)
graph = builder.compile()

5.6 StateGraph 和 Functional API 的区别 ​

LangGraph 官方有两套主流写法：

  • Graph API / StateGraph
  • Functional API

理解方式：

  • StateGraph：更显式，适合图式编排
  • Functional API：更自然，适合用普通函数控制流

重点：如果你想要“看见流程图”，优先用 StateGraph。

六、Persistence、Checkpointer 与线程 ​

6.1 什么是 Persistence ​

LangGraph 官方 Persistence 文档明确指出：

  • LangGraph 内置了 持久化层

它会在每一步执行时保存状态快照。

6.2 为什么 Persistence 重要 ​

Persistence 是以下能力的基础：

  • Durable execution
  • Human-in-the-loop
  • Time travel debugging
  • Fault tolerance

6.3 Checkpointer 是什么 ​

Checkpointer 用来保存图的状态。

开发时常见：

  • InMemorySaver

生产时常见：

  • PostgresSaver

6.4 Thread 的作用 ​

官方文档里非常强调：

  • 如果你用了 checkpointer，就必须提供 thread_id

因为 thread_id 是恢复执行和读取状态的关键。

6.5 InMemorySaver 示例 ​

python
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.graph import StateGraph, MessagesState, START, END


def bot(state: MessagesState):
    return {"messages": [{"role": "ai", "content": "收到"}]}


builder = StateGraph(MessagesState)
builder.add_node("bot", bot)
builder.add_edge(START, "bot")
builder.add_edge("bot", END)

graph = builder.compile(checkpointer=InMemorySaver())

config = {
    "configurable": {
        "thread_id": "thread-1"
    }
}

result = graph.invoke(
    {"messages": [{"role": "user", "content": "你好"}]},
    config=config,
)

print(result)

6.6 Durable Execution 是什么 ​

LangGraph 官方 Durable Execution 文档强调：

  • 只要你用了 checkpointer，就已经具备 durable execution 的基础

它可以让工作流：

  • 中断后恢复
  • 失败后继续
  • 长时间暂停后再继续

6.7 Durable Execution 的关键要求 ​

官方文档特别强调两个要求：

  1. 流程要尽量 deterministic
  2. 副作用和非确定性操作要包装好

例如：

  • 随机数
  • 外部 API 调用
  • 文件写入
  • 数据库写入

都要谨慎处理，避免恢复执行时重复副作用。

6.8 Durability 模式 ​

LangGraph 官方 Durable Execution 文档提到三种 durability 模式：

  • "exit"
  • "async"
  • "sync"

6.9 Durability 模式区别 ​

模式特点适用场景
exit只在执行退出时持久化性能优先
async异步持久化平衡性能与可靠性
sync同步持久化可靠性优先

6.10 获取状态示例 ​

LangGraph 的持久化能力也支持读取当前状态和历史状态。

python
snapshot = graph.get_state(config)
print(snapshot)

6.11 Persistence 学习重点 ​

  1. Checkpointer
  2. thread_id
  3. checkpoint
  4. durable execution
  5. 状态恢复

七、Interrupt 与 Human-in-the-loop ​

7.1 Interrupt 是什么 ​

interrupt() 可以让图在任意位置暂停，等待外部输入。

这在以下场景非常重要：

  • 人工审批
  • 表单确认
  • 高风险操作确认
  • 数据修改前审核

7.2 一个最小 Interrupt 示例 ​

python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.types import interrupt
from langgraph.checkpoint.memory import InMemorySaver


class State(TypedDict):
    action: str
    approved: bool


def approval_node(state: State):
    # 暂停图执行，并把审批信息返回给外部
    approved = interrupt({
        "question": "是否批准执行该操作？",
        "action": state["action"],
    })
    return {"approved": approved}


builder = StateGraph(State)
builder.add_node("approval", approval_node)
builder.add_edge(START, "approval")
builder.add_edge("approval", END)

graph = builder.compile(checkpointer=InMemorySaver())

7.3 Interrupt 的运行逻辑 ​

官方文档里强调：

  • 调用 interrupt() 后，图会暂停
  • 当前状态会被保存
  • 之后你可以通过 Command 恢复执行

7.4 使用 Command 恢复示例 ​

python
from langgraph.types import Command

config = {"configurable": {"thread_id": "approval-1"}}

# 第一次执行，会在 interrupt 处暂停
initial = graph.invoke({"action": "删除用户"}, config=config)
print(initial)

# 继续执行，并传入人工审批结果
result = graph.invoke(Command(resume=True), config=config)
print(result)

7.5 人工审批场景示例 ​

典型业务场景：

  • 删除数据库记录前要求审批
  • 调用支付接口前要求审批
  • 发送正式邮件前要求审批

7.6 Interrupt 的重要规则 ​

LangGraph 官方 Interrupt 文档特别强调：

  • 不要把 interrupt() 包在 try/catch 里
  • 传给 interrupt() 的内容要尽量可序列化
  • 恢复执行时，节点会重新执行，不是从某一行继续

重点：恢复执行不是“从上次那一行接着跑”，而是“从合适的恢复点重放执行”。

7.7 静态中断 ​

除了动态的 interrupt()，LangGraph 还支持：

  • interruptBefore
  • interruptAfter

它们可以在编译时设置。

但官方也提醒：

  • 真正的人类介入场景，更推荐使用 interrupt()

7.8 Human-in-the-loop 的本质 ​

LangGraph 里的人类介入，本质上是：

  • 图先跑
  • 跑到关键点暂停
  • 人给决策
  • 图继续跑

八、Memory、Store 与长期状态 ​

8.1 LangGraph 中的记忆分两类 ​

理解 LangGraph 的记忆一定要分清两类：

  • 线程内状态 / checkpoint 状态
  • 跨线程长期存储 / store

8.2 Checkpointer 和 Store 的区别 ​

类型作用生命周期
Checkpointer保存线程执行状态单线程
Store保存长期记忆跨线程 / 跨会话

8.3 InMemoryStore 示例 ​

python
from langgraph.store.memory import InMemoryStore

store = InMemoryStore()

store.put(("users",), "u_1001", {
    "name": "小明",
    "city": "上海",
    "preference": "喜欢简洁回答"
})

item = store.get(("users",), "u_1001")
print(item.value)

8.4 在图中使用 Store ​

python
from typing import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.store.memory import InMemoryStore


class State(TypedDict):
    user_id: str
    user_info: dict


store = InMemoryStore()
store.put(("users",), "u_1001", {"name": "小明", "city": "上海"})


def load_user_info(state: State, runtime):
    item = runtime.store.get(("users",), state["user_id"])
    return {"user_info": item.value if item else {}}


builder = StateGraph(State)
builder.add_node("load_user_info", load_user_info)
builder.add_edge(START, "load_user_info")
builder.add_edge("load_user_info", END)

graph = builder.compile(store=store)

8.5 语义搜索长期记忆 ​

LangGraph 官方 Persistence / Memory 文档提到：

  • Store 还可以支持 semantic search

这意味着长期记忆不只是精确 key-value 查找，还可以做语义检索。

8.6 生产环境存储方案 ​

开发环境常见：

  • InMemorySaver
  • InMemoryStore

生产环境常见：

  • PostgresSaver
  • PostgresStore

8.7 为什么 LangGraph 的 Memory 很重要 ​

因为 LangGraph 的记忆能力是：

  • 和工作流状态强绑定
  • 和持久化层强绑定
  • 和 Human-in-the-loop 强绑定

这和很多“只拼聊天历史”的简单方案完全不一样。

九、Functional API、Streaming 与生产能力 ​

9.1 Functional API 是什么 ​

LangGraph 官方专门提供了 Functional API，它适合：

  • 不想显式构建图
  • 想保留普通 Python 函数控制流
  • 但又想要持久化、记忆、中断、流式输出

9.2 Functional API 的两个核心装饰器 ​

  • @entrypoint
  • @task

9.3 Functional API 示例 ​

python
from langgraph.func import entrypoint, task
from langgraph.checkpoint.memory import InMemorySaver

checkpointer = InMemorySaver()


@task
def add_one(x: int) -> int:
    return x + 1


@entrypoint(checkpointer=checkpointer)
def workflow(inputs: dict) -> int:
    value = inputs["value"]
    a = add_one(value).result()
    b = add_one(a).result()
    return b


result = workflow.invoke({"value": 1}, config={"configurable": {"thread_id": "func-1"}})
print(result)

9.4 Graph API 和 Functional API 怎么选 ​

  • 复杂图结构明显：优先 StateGraph
  • 已有大量普通函数逻辑：优先 Functional API

9.5 Streaming 是什么 ​

Streaming 指的是图执行过程中：

  • 流式返回 token
  • 流式返回状态变化
  • 流式返回中断信息

9.6 图流式执行示例 ​

python
for chunk in graph.stream(
    {"messages": [{"role": "user", "content": "hi"}]},
    config={"configurable": {"thread_id": "stream-1"}},
):
    print(chunk)

9.7 前端实时展示场景 ​

LangGraph 官方 Frontend 文档提到：

  • 你可以把每个节点输出实时渲染到前端

适合：

  • 多步骤推理可视化
  • Agent 执行过程展示
  • 调试工具调用链

9.8 生产中的几个关键能力 ​

LangGraph 生产使用时，最关键的是：

  • Persistence
  • Streaming
  • Human-in-the-loop
  • Memory
  • Observability

9.9 可观测性 ​

LangGraph 在实际生产中通常要配合 LangSmith。

它能帮助你：

  • 追踪图执行路径
  • 看到每一步状态变化
  • 定位哪一节点失败
  • 分析延迟和成本

9.10 生产能力学习重点 ​

  1. StateGraph
  2. Functional API

十、LangGraph 与 LangChain 的关系、面试重点与最佳实践 ​

10.1 LangGraph 与 LangChain 的关系总结 ​

你可以这样理解：

  • LangChain 是“更容易上手的 Agent 框架”
  • LangGraph 是“更底层、更强控制力的工作流编排框架”

10.2 什么时候用 LangChain ​

适合：

  • 快速搭建一个 Agent
  • 快速做 Tool calling
  • 快速做 RAG Agent

10.3 什么时候用 LangGraph ​

适合：

  • 复杂工作流
  • 多步状态流转
  • 强控制 Agent
  • 持久化恢复执行
  • 人工审批

10.4 高频面试题 ​

  1. LangGraph 是什么
  2. LangGraph 和 LangChain 的区别
  3. StateGraph 的核心组成是什么
  4. 什么是 thread_id
  5. 什么是 checkpoint
  6. 什么是 durable execution
  7. Interrupt 是什么
  8. 为什么 Human-in-the-loop 需要 Persistence
  9. Checkpointer 和 Store 的区别
  10. Graph API 和 Functional API 的区别

10.5 最佳实践 ​

  • 先用 LangChain 快速起步，再决定是否下沉到 LangGraph
  • 复杂流程先设计状态，再设计节点
  • 所有需要恢复执行的流程都加 checkpointer
  • 生产环境不要只用内存存储
  • 高风险节点优先设计人工审批
  • 副作用操作要特别谨慎
  • 所有关键图执行都接入 LangSmith

10.6 一句话总结 ​

LangGraph 的主线可以概括为：

  • 用 State 描述工作流上下文
  • 用 Node 和 Edge 编排流程
  • 用 Checkpointer 保存状态
  • 用 Interrupt 支持人工介入
  • 用 Store 管长期记忆
  • 用 Streaming 和 LangSmith 支撑生产应用
